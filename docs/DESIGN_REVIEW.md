# League of Idea 设计评审与演进建议

本文记录对初版 PRD 与 v0.1 代码的评审结论。目标是先做成一个可靠、可解释、成本可控的终端工具，再扩展为更完整的 idea 研究系统。

## MVP 定义

MVP 应完成一条可验证的闭环：

1. 输入一个目标并生成候选 idea。
2. 按明确 rubric 做成对比较。
3. 更新 Elo 并保存每场证据。
4. 可选地进化高分 idea。
5. 输出、保存并能重新查看排行榜。

并发、SQLite、瑞士轮、多模型归因和 Web UI 暂不进入 MVP。它们都依赖稳定的比赛语义与会话数据格式，提前实现会放大返工成本。

## 已修正的关键问题

### 1. 默认成本过高

8 个初始 idea、3 轮、每轮进化 2 个 idea 时，全循环需要 144 次 LLM 调用；随机配对需要 35 次。因此默认策略改为 `random`，全循环保留给候选较少或需要完整比较的场景。运行前可用 `loi estimate` 估算调用次数。

### 2. 排名存在出生分污染

进化出的 child 原先直接继承 parent 的 Elo，会在没有获胜的情况下占据高位，也会向评分池凭空注入高分。现在所有新 idea 都从统一基线 1200 开始，靠比赛获得排名。

### 3. 付费结果可能因中断丢失

会话原先只在一整轮结束后保存。现在生成完成后、每场比赛后以及每轮结束后都会原子保存；异常会把会话标记为 `failed` 并记录错误信息。

### 4. provider 接口已发生变化

any-llm 当前推荐 `provider:model`，代码同时兼容旧的 `provider/model` 输入，并在内部拆成独立的 provider 与 model 参数。基础模型调用会做指数退避重试。

### 5. 裁判位置偏差

仅在 prompt 中声明“不要受顺序影响”并不可靠。当前版本会随机交换 A/B 展示顺序并记录实际位置；后续可增加双向裁判与一致性检查。

## 建议新增的模块

以下模块按优先级逐步加入，不建议一次性全部实现。

### P1：下一版应做

- `rubric.py`：把 novelty、feasibility、relevance 的定义、权重与版本从 prompt 中抽离。每个 Session 保存 rubric 版本，保证历史结果可解释。
- `resume.py`：根据 `status`、已完成轮次和已有 match 继续失败会话，并避免重复计分。
- `budget.py`：把调用次数、输入输出 token、估算费用和预算上限统一管理；到达上限时安全停止。
- `report.py`：除 Elo 外输出胜负证据、分歧较大的比赛、idea 谱系和 Markdown/JSON 报告。

### P2：质量与效率

- `pairing.py` 增加瑞士轮或基于不确定性的主动配对，减少无信息比赛。
- `judge.py` 增加双向裁判、平局、置信度和分维度评分。A/B 两次结论不一致时标记为争议，而不是强行判胜。
- `dedup.py` 增加语义去重；当前仅拦截规范化后完全相同的文本。
- 异步 runner 与 provider 级并发限制。并发只改变执行速度，不应改变 tournament 的业务语义。

### P3：规模化后再做

- SQLite 存储、数据迁移和查询索引。
- 多模型生成归因与裁判校准。
- Web UI 或多人协作。

## 数据模型建议

Elo 只表示在特定目标、rubric、裁判模型与比赛集合下的相对强弱，不应被解释为绝对质量。后续 Match 建议加入：

- rubric 版本与各维度分数；
- 裁判模型、调用时间、token 用量与成本；
- position swap 信息、置信度和是否争议；
- prompt 版本或哈希，支持复现实验。

Session 应继续保存完整运行配置、schema version、状态、时间戳和随机种子。若以后切到 SQLite，JSON 仍可作为稳定的导入导出格式。

## 测试策略

测试分三层：

1. 纯函数：Elo、配对、调用量估算。
2. 无网络集成测试：模拟 generator/judge，验证完整 tournament、落盘与失败恢复语义。
3. 少量真实 provider 冒烟测试：手动或 CI secret 环境运行，不作为普通单元测试的前置条件。

当前测试覆盖前两层的关键路径；真实 API 调用需要用户配置密钥后单独验证。
